# 0061 — Phase 3.5 Sub-Phase C: Badge earn loop

**Branch:** `claude/peaceful-thompson-AXsL9`. Draft PR.

## What this lands

Sub-Phase C of the Phase 3.5 roadmap: the Samaritan-threshold badge award machinery and the in-world badge-earned notification.

### Migration `0009_badge_earn_loop.sql`

- **Realtime publication**: adds `earned_badges` to `supabase_realtime` (idempotent DO block) so the client can subscribe to its own INSERT events through the existing RLS SELECT policy (`auth.uid() = user_id`).
- **`award_pending_badges(p_uid UUID) → TEXT[]`** — internal SECURITY DEFINER helper. Reads the user's `samaritan_corporate` and `samaritan_bitrunner` scores, then inserts every crossed-but-not-yet-owned tier row into `earned_badges` for both factions. Returns the newly-inserted keys. Cap is 10 tiers at +10 each → tier at 100. NOT granted to `authenticated`; only callable from within other SECURITY DEFINER functions.
- **`admin_grant_samaritan(target UUID, p_faction TEXT, p_amount INT) → TABLE(new_score INT, new_badges TEXT[])`** — admin-only RPC. Increments `profiles.samaritan_{corp|bitrunner}` by `p_amount` (capped at 100), calls `award_pending_badges`, returns the resulting score + any freshly-minted badge keys. Amount must be 1–100; faction must be `'corp'` or `'br'`. Granted to `authenticated` but gated by `is_admin(auth.uid())`.

### `apps/web/src/badge-notifications.ts` (new)

Supabase Realtime monitor for `earned_badges` INSERT events:
- `startBadgeMonitor()` — boots once, subscribes to `subscribeAuth` to create/tear-down the Realtime channel as the user signs in/out. Idempotent.
- On INSERT for the current user: calls `profile.incrementUnacknowledged()` (so the `!` dot on the head label lights up immediately) and dispatches `'bitrunners:badge-earned'` with `{ badgeKey, earnedAt }`.
- Exports `BADGE_EARNED_EVENT` and `BadgeEarnedDetail` for consumers.

### `apps/web/src/BadgeToast.tsx` (new)

In-world badge-earned notification stack:
- Listens for `'bitrunners:badge-earned'` events.
- Renders a right-side column of toasts, each with the badge glyph (faction-tinted), tier name, faction label, and "badge unlocked" marker.
- Auto-dismisses after 4 s; cleans up timers on unmount.
- `prefers-reduced-motion`: animation removed.
- Multiple badges (e.g. skipping tiers via a large admin grant) stack vertically.

### `apps/web/src/profile.ts`

- Added `incrementUnacknowledged()` — counterpart to the existing `decrementUnacknowledged()`. Called by `badge-notifications.ts` when a Realtime INSERT fires so the `!` dot on the head label flips without waiting for a full `refreshIdentity()` round-trip.

### `apps/web/src/supabase.ts`

- Added `adminGrantSamaritan(target, faction, amount)` wrapper for the new RPC. Returns `{ newScore, newBadges }` on success or `{ error }` on failure.

### `apps/web/src/AdminConsole.tsx`

- Added **`SamaritanGrant`** sub-component inside `UserEditor`. Shows a faction select (`corp` / `br`), a numeric amount input (1–100), and a `[ samaritan ]` button. On success, shows the new score and any newly awarded badge keys (e.g. `score → 30 · new: corp:wood, corp:stone, corp:bronze ✓`). This gives the owner a test path without waiting for missions or NPC interactions to be built.

### `apps/web/src/App.tsx`

- Imports and calls `startBadgeMonitor()` at module level (alongside `startIdentity()`).
- Mounts `<BadgeToast />` inside `Game`.

### `apps/web/src/style.css`

- New `.badge-toast-stack`, `.badge-toast`, `.badge-toast-glyph`, `.badge-toast-body`, `.badge-toast-tier`, `.badge-toast-faction`, `.badge-toast-label` rules. Slide-in animation from right; `prefers-reduced-motion` strips it.

## Deployment notes

1. **Owner: run migration `0009` in Supabase.** Without it:
   - `admin_grant_samaritan` doesn't exist → the admin grant button errors.
   - `earned_badges` is not in the Realtime publication → badge-earned push notifications silently don't fire (the UI still works, badges just don't toast in real-time; `get_my_badges` polling on sign-in will still reflect them).
2. **Prereq: migration `0007` must be applied first** (Sub-Phase B, which creates `earned_badges`, `samaritan_corporate`, `samaritan_bitrunner`).
3. No server change → Pages-only deploy. No Fly redeploy needed.

## What's deferred

- Actual Samaritan score sources: missions (Sub-Phase G), NPC interactions (Admin NPC first encounter, Company NPC). Until those ship, the only path to earn badges is via admin grant (test path).
- Realtime badge push requires `0009` to be applied and an active Supabase session. Signed-out / unconfigured Supabase still works — badges just don't toast live (they appear in `get_my_badges` on the next sign-in).
- Client `subscribeBadgeEarned` is not exported from `supabase.ts` as a standalone — it's internal to `badge-notifications.ts`. This is fine because the monitor is the only consumer.
